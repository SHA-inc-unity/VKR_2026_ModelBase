## Dataset jobs (редизайн «длительные операции — это jobs»)

Все длительные операции датасета (ingest, market_watch,
detect_anomalies, compute_features, clean_apply, export, import_csv,
upsert_ohlcv) принимаются как **jobs** —
синхронные RPC-команды только инициируют запись в `dataset_jobs` и сразу
возвращают `{ job_id, status: 'queued', conflict?: { existing_job_id } }`.
Реальная работа идёт в `DatasetJobRunner` (BackgroundService), а прогресс
публикуется в `events.data.dataset.job.progress` / `…job.completed`.

Контракт топиков:

- `cmd.data.dataset.jobs.start` — создать job (валидация payload, вставка
  в `queued`; если `(target_table, conflict_class)` уже занят — возвращает
  `conflict_existing_job_id`).
- `cmd.data.dataset.jobs.cancel` — выставить `cancel_requested=true`;
  бегущая job ловит CancellationToken на следующем `await` и завершается
  как `canceled`.
- `cmd.data.dataset.jobs.get` / `cmd.data.dataset.jobs.list` — read-only
  view (`active` | `terminal` | конкретный статус), включая `stages[]` и
  `subtasks[]`.

Семантика статусов: `queued | running | succeeded | failed | canceled |
skipped` (`skipped` — когда работы нет, например, нет недостающих свечей).

Ключевые компоненты:

| Файл | Роль |
| ---- | ---- |
| `Jobs/DatasetJobRunner.cs` | BackgroundService-шедулер. На старте сначала делает `WaitForSchemaReadyAsync()` (retry-loop вокруг `EnsureSchemaAsync`, без блокировки `app.Run()`), затем выполняет startup-recovery: `FailInvalidQueuedAsync()` для старых broken queued rows и `ReclaimOrphansAsync()` (`running → failed` с `error_code='service_restart'`). Дальше в цикле выбирает **FIFO** `queued`-задания через `PickQueuedAsync(ORDER BY created_at)`, проверяет per-type `SemaphoreSlim` + `JobLockManager`, делает атомарный CAS `TryAcquireRunningAsync(version)` и диспатчит handler через `Task.Run`. Heartbeat пишется каждые 5 с. По завершении публикуется `events.data.dataset.job.completed` и освобождаются lock + slot. **Дополнительные ограничения для ingest**: у каждой биржи свой пул из 4 одновременных ingest-job'ов (`bybit` / `binance` / `kraken`) и свой single-slot heavy gate для `1m`/`3m`, так что одна биржа больше не забивает общий ingest-runner целиком. Отдельно есть global cap `1` для `market_watch`, чтобы live-worker оставался queue-visible singleton. Для Kraken эти 4 job-слота не означают 4 бесконтрольных REST burst-а: реальный HTTP budget теперь режется внутри `KrakenApiClient` process-local limiter-ом. Для Binance те же 4 scheduler-slot'а теперь дополнительно отделены от реального REST budget process-local limiter-ом внутри `BinanceApiClient`, поэтому несколько binance jobs не должны снова сжигать IP баном весь ingress. `PublishCompleted` включает финальный snapshot `overall_progress` + `stage_*` counters, а не только `completed` rows. |
| `Jobs/MarketWatchBootstrapService.cs` | Отдельный hosted service, который при `DataService:MarketWatch:Enabled=true` периодически гарантирует один active job `type=market_watch` с фиксированным `params_hash = market_watch:singleton:v1`. Не трогает исторические таблицы; ставит в очередь только singleton job на target table `market_watch_live`. |
| `Jobs/MarketWatchJobHandler.cs` | Перпетуальный handler для live-overlay рынка. На старте раскрывает symbol universe через `MarketDataClientFactory`, затем открывает websocket price streams (`Binance.Net`, `Bybit.Net`, `KrakenExchange.Net`), локально агрегирует current candles по всем настроенным timeframe и раз в `FlushIntervalMs` сохраняет snapshots в `market_watch_live`. Исторические dataset tables не мутируются. |
| `Jobs/JobLockManager.cs` | Process-local карта `(conflict_class, target_table) → job_id` (`ConcurrentDictionary<string,byte>`, ключ `"{conflictClass}::{table ?? "*"}"`). `mutating_table` блокирует запись по таблице, `read_heavy` ограничивается только лимитом на тип, `market_watch` резервирует singleton live table `market_watch_live`, `external_io` остаётся под ingest semantics. **Важно**: `target_table=null` сворачивается в общий ключ `*` per-conflict_class — поэтому ingest-job'ы обязаны иметь непустой `target_table` (см. `KafkaConsumerService.Jobs.cs`), иначе все они сериализуются в один лок. |
| `Jobs/IDatasetJobHandler.cs` | Контракт handler-а + `JobContext` (репортер прогресса, проверка отмены, helpers `StartStageAsync` / `EndStageAsync` / `AddSubtaskAsync` / `UpdateSubtaskAsync`). `JobContext.ReportAsync` теперь ведёт два слоя прогресса: weighted overall `progress/overall_progress` и current-stage snapshot `stage_progress`, `stage_total`, `stage_completed`, `stage_failed`, `stage_skipped`. Всё это публикуется в `events.data.dataset.job.progress` best-effort и одновременно пишется в `dataset_jobs`, чтобы `jobs.list/get` переживали missed SSE. |
| `Jobs/JobHandlers.cs` | 7 реализаций `IDatasetJobHandler`. `IngestJobHandler` пишет stages `prepare → fetch (klines/funding/oi parallel + RSI inline) → upsert → compute_features`; выбирает `maxParallel` по exchange+TF: тяжёлые `1m`/`3m` Bybit и Binance backfills получают более широкий in-job fan-out `6`, потому что scheduler всё равно держит один heavy-slot на биржу, а реальный REST budget дальше режется соответствующим client-side limiter-ом (`BybitRateLimiter` / `BinanceRateLimiter`). Kraken heavy path остаётся на `2`. Для остальных TF используется `8`. Handler читает `params.exchange` (default `bybit`) и делегирует fetch в `MarketDataClientFactory`: `bybit` и `binance` дают полный klines/funding/OI pipeline, `kraken` — OHLC-only client с локальной агрегацией неподдерживаемых TF и пустыми funding/OI. Для ingest handler теперь расходятся два смысла прогресса: overall `progress` остаётся weighted pipeline percent, а `fetch_klines` / `fetch_oi` дополнительно публикуют честный page-based `stage_progress` + `stage_total/completed`, поэтому UI может отдельно показывать текущий этап и общий pipeline. Для Kraken handler заранее клипует requested window к реально достижимой lookback-history (последние 720 base candles); если окно полностью вне reachable history, job завершается как no-op, а не как runtime error. `CleanApplyJobHandler` открывает один `NpgsqlConnection` и применяет op-список (`drop_duplicates`, `fix_ohlc`, `fill_zero_streaks`). `DetectAnomaliesJobHandler` пробегает 10 detection stages. `ComputeFeaturesJobHandler` — обёртка над репозиторием. `Export/ImportCsv/UpsertOhlcv` — заглушки; legacy команды по-прежнему доступны. |
| `Database/DatasetJobsRepository.cs` | Read-only выборки + DDL `dataset_jobs` (теперь не только `progress`/`stage`, но и `stage_progress`, `stage_total`, `stage_completed`, `stage_failed`, `stage_skipped`, чтобы live UI не зависел только от SSE). Метод `MapJobPublic` экспонирует маппер для мутатора. **Schema guard**: `_schemaReady` флаг + публичное свойство `SchemaReady`; `StartAsync` бросает `InvalidOperationException` если `EnsureSchemaAsync` не завершился — Kafka-handler делает быстрый precheck `SchemaReady` и возвращает `{ error, code: "schema_not_ready" }` вместо таймаута. **Active-job dedup**: единый источник правды — partial unique INDEX `uq_dataset_jobs_active_params ON dataset_jobs (params_hash) WHERE status IN ('queued','running')`; `INSERT … ON CONFLICT (params_hash) WHERE status IN ('queued','running') DO NOTHING` использует **index inference** (форма `ON CONFLICT ON CONSTRAINT <name>` для partial-индексов не работает и падает на `42P10`). Идемпотентно на чистой и существующей БД. |
| `Database/DatasetJobsMutator.cs` | Все мутации: `TryAcquireRunningAsync` (CAS по `version`), `UpdateProgressAsync`, `HeartbeatAsync`, `FinishAsync`, `PickQueuedAsync`, `IsCancelRequestedAsync`, `ReclaimOrphansAsync`, `FailInvalidQueuedAsync` (мягко переводит в `failed` старые structurally-broken queued rows, не трогая пользовательские данные), `Add/UpdateSubtaskAsync`, `Start/EndStageAsync` (последний апсертит `custom_metrics`). `UpdateProgressAsync` теперь пишет и overall, и current-stage counters одним апдейтом, а не только один смешанный `progress`. |
| `Database/MarketWatchRepository.cs` | Отдельное хранилище live-overlay состояния. Идемпотентно создаёт `market_watch_live(exchange, symbol, last_price, last_price_ts, candles_json, updated_at)` и upsert-ит один snapshot на `(exchange, symbol)`. Все текущие timeframe candles лежат в одном `jsonb`, чтобы worker не писал отдельную строку на каждый TF каждую секунду. |
| `Bybit/BybitRateLimiter.cs` | Process-local token-bucket (default 96 r/s = 80 % от Bybit IP-лимита) + `BybitApiException(retCode, retMsg)`. `BybitApiClient.GetJsonAsync` ждёт токен, делает экспоненциальный backoff с jitter на 429/5xx, ловит `retCode != 0` и пробрасывает без retry. |
| `Database/DatasetRepository.cs` | Метод `GetCoverageRangeAsync(table, startMs, endMs, stepMs)` — coverage внутри окна (rows-in-range + expected через `generate_series`). Используется в `HandleCoverageAsync`, когда передан явный `[start_ms, end_ms]`. |

## src/DataService.API/

Единственный проект (ASP.NET Core Minimal API / Controllers). Clean Architecture пока не нужна — весь код здесь.

### Program.cs

Точка входа: регистрация DI, Serilog, Kafka consumer, job-runner, `MarketWatchBootstrapService`, MinIO, healthchecks и роутинга. В DI теперь регистрируются typed `HttpClient`-ы для `BybitApiClient`, `BinanceApiClient`, `KrakenApiClient`, singleton `MarketDataClientFactory`, `MarketWatchRepository` и process-local rate limiters для `Bybit`, `Binance`, `Kraken`. Важный lifecycle-инвариант: `Program` **не** ждёт `EnsureSchemaAsync()` до `app.Run()`; HTTP `/health` и `/ready` поднимаются сразу, а bootstrap jobs-схемы уходит в `DatasetJobRunner`.

Имя колонки `timestamp_utc` в SELECT заменяется на вычисляемое `timestamp_ms` (контракт Kafka), идентификатор таблицы проходит `Safe()`. Используется `HandleExportAsync` в связке с `System.IO.Pipelines`. `BulkUpsertAsync` (UNNEST + `ON CONFLICT (timestamp_utc) DO UPDATE`, батчи по `DatasetConstants.UpsertBatchSize`), `DeleteRowsAsync(table, startMs?, endMs?)` — если оба timestamp-а `null` → `TRUNCATE TABLE` (возвращает число строк до очистки), иначе `DELETE WHERE timestamp_utc IN [start, end]`; для отсутствующей таблицы возвращает `0` без исключения. **Anomaly Inspect:** `GetColumnStatsAsync(table, columnFilter?, countOnly?)` — возвращает `ColumnStatsResult(TotalRows, Columns[ColumnStat(Name, Dtype, NonNull, Min, Max, Mean, Std)])` одним SQL-запросом; `columnFilter` (список имён) ограничивает набор колонок; `countOnly=true` убирает `MIN/MAX/AVG/STDDEV` — только `COUNT(*)` + per-column `COUNT("col")::bigint AS nn_{col}`; имена идентификаторов проходят `Safe()`-гард. `GetColumnHistogramAsync(table, column, buckets)` — возвращает `HistogramResult(Column, Min, Max, Buckets[HistogramBucket(RangeStart, RangeEnd, Count)])`; buckets clamp 2..500; `width_bucket(col, lo, hi + (hi-lo)*1e-9, buckets)` с обрезкой индекса 0..buckets-1; пустые buckets дозаполняются нулями; при hi ≤ lo — один bucket с полным count; для полностью null-колонки — пустой массив buckets. Для отсутствующей таблицы оба метода возвращают `null`. |

Каждый метод возвращает `IReadOnlyList<AnomalyRow(TsMs, AnomalyType, Severity, Column, Value, Details)>`. **Apply**: `ApplyDropDuplicatesAsync` (DELETE USING ROW_NUMBER on ctid keep first), `ApplyFixOhlcAsync` (сначала проверяет `information_schema.columns` через открытый `conn`, возвращает `0` если отсутствует хотя бы один из `_ohlcCols`; иначе UPDATE high/low = GREATEST/LEAST), `ApplyFillZeroStreakAsync(table, column, conn)` (CTE running SUM grp + FIRST_VALUE forward fill UPDATE), `ApplyDeleteByTimestampsAsync` (NpgsqlParameter Array TimestampTz), `ApplyFillGapsAsync` (`generate_series` + LATERAL prv/nxt + per-column expressions: `linear` → `(prv)."col" + ((nxt)."col"-(prv)."col")*frac`; `forward_fill` → `COALESCE((prv)."col",(nxt)."col")`; скобки вокруг `prv`/`nxt` **обязательны** — без них PostgreSQL интерпретирует их как псевдонимы таблиц и падает с "missing FROM-clause entry"; ON CONFLICT (timestamp_utc) DO NOTHING). **Lock**: `AcquireApplyLockAsync` возвращает live `NpgsqlConnection` под `pg_advisory_lock(hash_record_extended(ROW(@t), 0))`; `ReleaseApplyLockAsync` снимает блокировку и dispose-ит соединение (вызывается в `finally`). **Audit**: `EnsureAuditLogAsync` создаёт `dataset_audit_log(id SERIAL PK, table_name, operation, params JSONB, rows_affected INT, applied_at TIMESTAMPTZ DEFAULT now())` + index. `WriteAuditLogAsync(table, op, paramsJson, rowsAffected)` → `int audit_id`. |

Постраничная навигация по полному списку — через `{ page, page_size }` (page_size clamp 1..5000) → `rows` несёт slice. Когда `total > 200`, полный отчёт (все строки) сохраняется в MinIO `reports/anomaly_<guid>.json` и presigned URL возвращается в `report_url` (60 мин TTL). UI никогда не получает unbounded JSON-массив. |
| `cmd.data.dataset.clean.preview` | req/reply | Превью операций очистки. Ответ: `{ table, counts: { drop_duplicates, fix_ohlc, fill_zero_streaks, delete_by_timestamps, fill_gaps } }`. |
| `cmd.data.dataset.clean.apply` | req/reply | Применение очистки в БД. Требует `confirm: true`. `pg_advisory_lock(hash_record_extended(ROW(@t), 0))` ключ по имени таблицы. Порядок: drop_duplicates → fix_ohlc → fill_zero_streaks (loop columns с `try/except PostgresException` для отсутствующих) → delete_by_timestamps → fill_gaps (`linear` или `forward_fill`). Аудит: `EnsureAuditLogAsync` создаёт `dataset_audit_log(id SERIAL PK, table_name, operation, params JSONB, rows_affected INT, applied_at TIMESTAMPTZ)`; `WriteAuditLogAsync` возвращает `audit_id`. **Lock leak fix**: `WriteAuditLogAsync` открывает собственное соединение (не переиспользует `conn` из `AcquireApplyLockAsync`), поэтому пара audit + `ReleaseApplyLockAsync` обёрнута в `try/finally` — advisory-lock гарантированно снимается даже если INSERT в `dataset_audit_log` бросит исключение. Ответ: `{ table, audit_id, rows_affected: {…}, total }`. |
| `cmd.data.dataset.compute_features` | req/reply | Идемпотентный SQL-пересчёт 27 feature-колонок через window functions (`LAG`, `AVG/STDDEV_POP/MIN/MAX OVER PARTITION BY symbol, timeframe`). Перед `UPDATE` — `ALTER TABLE … ADD COLUMN IF NOT EXISTS` для каждой фичи (nullable `double precision`). Обе SQL-команды (ALTER и UPDATE) выполняются с `commandTimeout: 0` (бесконечный Npgsql-таймаут) — для 1m-таблиц с >2.5 млн строк дефолтных 30 с недостаточно (остальные запросы репозитория сохраняют дефолтный таймаут). Payload `{ table }` → `{ status, table, rows_updated }`. Автоматически вызывается в ingest как отдельный прогресс-этап `compute_features`. |
